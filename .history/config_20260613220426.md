# Configuración del sistema — Índice de Riesgo de Deserción (IRD)

Este documento contiene el JSON que debe cargarse como **texto plano** en la
entrada de la **Db Config** (`https://ingenieria.campus.mdp.edu.ar/mod/data/view.php?id=54785`,
`d=17`), junto con las aclaraciones necesarias para entenderlo y completarlo.

---

## 1. Aclaraciones generales

### 1.1 Cmids actualizados

- **Db Histórico**: cmid `54788` → ya incluido en el JSON.
- **Db Config**: cmid `54785`. **No se incluye en el JSON** porque sería una
  referencia circular (es justamente donde se guarda este JSON). Esta
  referencia debe quedar hardcodeada en el código, no en la configuración.

### 1.2 Cambio de metodología: obtención de datos de las encuestas (Feedback) vía CSV

Para las actividades **Feedback** (encuesta inicial y cuatrimestral) ya **no**
se hace scraping del HTML de `show_entries.php`. En su lugar, el script hace
`fetch` directo al endpoint de exportación CSV:

```js
const url = `${BASE_URL}/mod/feedback/show_entries.php?sesskey=${M.cfg.sesskey}&download=csv&id=${CMID_FEEDBACK}`;
const response = await fetch(url, { credentials: "include" });
const csvText = await response.text();
```

- `CMID_FEEDBACK` para **Encuesta Inicial** = `54062`.
- `CMID_FEEDBACK` para **Encuesta Cuatrimestral** = `54284` (cambió respecto al
  cmid anterior, `54067` — la actividad fue recreada/clonada).
- El CSV sigue **RFC 4180** (campos con comas/comillas/saltos de línea van entre
  comillas dobles). **No usar `split(',')`**, usar un parser con soporte de
  _quoting_.
- Esta decisión aplica **solo a Feedback**. Las actividades **Database** (Db
  Solicitudes, Db Histórico, Db 1er Año, etc.) siguen funcionando con el
  mecanismo de POST a `edit.php` + mapeo `label → field_X` ya documentado.

### 1.3 Identificación del alumno (problema abierto)

El CSV de Feedback incluye `Nombre completo del usuario` y `Dirección de
correo`, pero **no** el ID numérico de Moodle del usuario. El cruce CSV ↔ ID
de Moodle se hace **por email** (sin riesgo de colisión, según lo definido).

Sin embargo, el **ID de Moodle igual debe adquirirse** por otro medio (p. ej.
scraping del listado de participantes), porque se necesita para identificar al
alumno en las Db (Solicitudes, Histórico, 1er Año, etc.). Es decir, el flujo
combina:

- **CSV** → todas las respuestas de las encuestas (cruzadas por email).
- **Scraping del listado de participantes** → ID de Moodle de cada alumno
  (cruzado también por email/nombre).

Esto queda documentado como decisión de implementación; **no requiere campos
adicionales en este JSON**, salvo que el equipo decida lo contrario.

### 1.4 Sobre los `id` de las preguntas

Dado el cambio a exportación CSV, el campo `id` de cada pregunta en este JSON
es el **texto exacto del encabezado de columna que Moodle genera en el CSV**
(que normalmente coincide con el texto/label de la pregunta en el Feedback).
**Verificar cada uno contra un CSV real exportado**, ya que los textos
puestos aquí provienen del documento de planificación y podrían no coincidir
100% (mayúsculas, tildes, signos de puntuación, etc.).

### 1.5 Preguntas especiales — estado

- `id_percepcion_riesgo` ("¿Crees que estás en riesgo de abandonar la
  carrera?"): **CONFIRMADO** — funciona puramente como bandera automática, sin
  β/X propio en el cálculo del IRD.
- `id_anio_ingreso`: confirmado, label = "¿En qué año ingresaste a la
  carrera?".
- `id_egreso`: **sigue siendo una pregunta de encuesta** (no se resolvió aún
  cuál es su label exacto). Queda como `PENDIENTE` — falta identificar la
  pregunta y su columna en el CSV de la encuesta cuatrimestral.

### 1.6 Resumen de pendientes reales (lo único que falta para cerrar el JSON)

1. Confirmar/ajustar el `id` (encabezado CSV) de `id_egreso`.
2. Verificar, contra un CSV real exportado de ambas encuestas, que todos los
   `id` de las preguntas (encabezados de columna) coinciden exactamente con
   los strings de este JSON.

Todo lo demás (β, valores X, umbrales, cmids/dataids, fórmulas que van en
código) ya está definido.
